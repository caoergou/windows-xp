# 提议: 解密体验优化

*基于设计文档与实际数据的全面审查*

## 一、核心发现: 设计文档与实际数据存在严重不一致

### 1.1 puzzle_hints.json 与设计文档的密码/线索完全矛盾

这是最紧急的问题。`puzzle_hints.json` 中的密码和线索与 `docs/解密.md` 和 `docs/设计.md` 的设计完全不同:

| 谜题 | 设计文档密码 | puzzle_hints.json 密码 | 设计文档线索来源 | puzzle_hints.json 线索来源 |
|------|-------------|----------------------|----------------|--------------------------|
| QQ密码 | `19990315`(夏灯生日) | `19990815`(林晓宇生日) | 父亲遗言"密码是你生日" | 桌面便签上林晓宇生日 |
| 相册密码 | `28125400`(f/2.8-1/125-ISO400) | `2.8-250-800` | 群聊"光之公式"讨论 | "山间记忆"相册最后一张照片EXIF |
| 父亲第一层 | `19990315`(夏灯生日) | `19990312`(陈默生日) | README.txt直接告知 | QQ聊天记录中陈默生日 |
| 加密日志 | `camera3rdeye` | `1.8-60-200` | QQ空间隐写文字 | 林晓宇珍视的山间照片参数 |
| 父亲第二层 | `20151123`(签署虚假证明日) | `20160115`(林晓宇坠亡日) | README.txt"灯熄灭的日子" | 父亲日志中的案件日期 |

**实际游戏数据中的密码**（filesystem.json / encrypted_diary.json / albums.json）与设计文档一致:

- 给小灯文件夹: `19990315`
- 爸爸的信: `20151123`
- 加密相册: `28125400`
- 加密日志: `camera3rdeye`

**结论**: `puzzle_hints.json` 是一个早期版本的残留，与当前设计和实际游戏数据完全脱节。必须重写。

### 1.2 EXIF数据不一致

设计文档中林晓宇使用 Canon EOS 550D，参数为 f/2.8, 1/125, ISO400。但实际 `photo_001_exif.json` 中:

- 相机: Sony DSC-W830
- 光圈: f/4.5
- 快门: 1/250
- ISO: 400

这意味着即使玩家按照设计意图查看EXIF数据，看到的参数也无法推导出密码 `28125400`。

## 二、逐谜题线索链分析

### 谜题0: QQ登录 (非谜题，直接提供)

**设计**: 密码 `19990315`，通过父亲遗言"密码是你生日"直接告知。

**实际数据**: `README.txt` 中明确写着"密码是你的生日: 19990315"。但这个文件在加密文件夹内部，玩家需要先解锁文件夹才能看到。

**问题**: 设计文档说QQ登录密码"直接提供给玩家（电脑开机后父亲遗言提及）"，但实际上开机后玩家如何得知密码？需要确认是否有桌面便签或开机提示实现了这一点。如果没有，玩家在游戏最开始就会卡住。

**建议**: 确保开机后桌面有明确的便签或提示文件，告知"密码是你的生日"。

### 谜题1: 林晓宇加密相册 (难度 2/5)

**线索链分析**:

```
铺垫: 群聊2015.11.20 林晓宇提到"f/2.8, 1/125, ISO400"（光之公式）
  |
发现: 相册描述提示"能定格时间的三个数字"+"看看照片属性"
  |
尝试: 玩家需要将摄影参数转化为数字组合
  |
解密: 输入 28125400
  |
奖励: 4张证据照片 + 引导去看加密日志
```

**问题**:

1. 从 "f/2.8, 1/125, ISO400" 到 "28125400" 的转换规则不明确。玩家可能尝试 "2.8125400"、"2.8-125-400"、"281254OO" 等多种变体。
2. 设计文档说"下载相册封面图片到本地，右键查看EXIF"，但实际相册封面为空（`coverImg: ""`），且 `photo_001_exif.json` 的参数（f/4.5, 1/250）与密码不匹配。
3. 相册描述中的提示文字（"能定格时间的三个数字"）在实际 `albums.json` 中不存在，只有 `name: "加密"` 和 `password: "28125400"`。

**建议**:

- 在 `albums.json` 的加密相册中添加 `description` 字段，包含提示文字
- 修正 `photo_001_exif.json` 使其与设计一致（Canon EOS 550D, f/2.8, 1/125, ISO400）
- 考虑在密码输入框的提示文字中暗示格式（如"请输入8位数字密码"）

### 谜题2: 父亲文件夹第一层 (难度 2/5)

**线索链分析**:

```
铺垫: 父亲遗言/便签提到"密码是你的生日"
  |
发现: 文件夹名"给小灯"暗示与夏灯有关
  |
尝试: 输入夏灯生日
  |
解密: 19990315
  |
奖励: 工作日志（大量误导证据）+ 网页截图 + 发现第二层加密文件夹
```

**评价**: 这是设计最清晰的谜题。线索直接、密码有情感意义、奖励内容丰富。

**问题**: `README.txt` 同时直接给出了第二层密码 `20151123`，这使得谜题5（父亲第二层）完全没有解密过程。玩家打开 README 就能看到两层密码。

**建议**: README.txt 不应直接给出第二层密码。应该只暗示"那个文件夹的密码是爸爸的灯熄灭的日子"，让玩家通过阅读工作日志推理出 2015.11.23 是夏父签署第一份虚假证明的日期。

### 谜题3: 林晓宇加密日志《第三只眼》(难度 3/5)

**线索链分析**:

```
铺垫1: 群聊2015.12.28 林晓宇提到"把信息藏在诗里"+"每行第三个字连起来读"
铺垫2: QQ空间个性签名 "Camera is the 3rd eye."
铺垫3: 相册描述引导"去看加密日志"
  |
发现: 日志预览区有隐藏的白色文字
  |
关键操作: 选中文字（Ctrl+A或鼠标拖选）
  |
发现: "Camera is the 3rd eye"
  |
解密: 输入 camera3rdeye
  |
奖励: 现代诗《第三只眼》+ 日记正文（陈默动机、真相反转）
```

**严重问题: 隐写文字机制未实现**

审查 `QZone.jsx` 代码发现:

- 加密日志解锁后，内容直接以纯文本渲染（第1135行: `<div className="blog-detail-content">{viewingItem.data.content}</div>`）
- 没有任何白色隐藏文字的实现
- 没有预览区域的概念（加密日志要么锁定，要么完全显示）
- `encrypted_diary.json` 中也没有 `preview` 字段（设计文档中有 `preview.visible` 和 `preview.hidden`）

这意味着设计文档中最精巧的解密机制——QQ空间隐写术——在代码中完全没有实现。当前的实现是: 玩家看到加密日志 -> 输入密码 -> 看到全部内容。但密码 `camera3rdeye` 的发现路径（通过选中隐藏文字）不存在。

**玩家实际体验**: 看到加密日志，不知道密码是什么，只能靠猜测或等提示系统给答案。而提示系统（puzzle_hints.json）给的密码是错误的 `1.8-60-200`。

**建议**:

1. 在加密日志的列表项或预览区域实现白色隐藏文字机制
2. 在 `encrypted_diary.json` 中添加 `preview` 字段
3. 实现选中文字时的视觉反馈（文字从不可见变为可见）
4. 这是整个解密系统中最有创意的设计，值得优先实现

### 谜题4: 现代诗藏头 (难度 4/5)

**线索链分析**:

```
铺垫: 群聊2015.12.28 "每行第三个字连起来读"
  |
发现: 解锁日志后看到诗歌，末尾有提示"（每行第三个字连起来读）"
  |
操作: 手动提取每行第3字
  |
结果: "我会被王虎杀死他在操作高考移民"
```

**评价**: 这个谜题设计精妙，但有一个关键问题——诗歌末尾直接写了"（每行第三个字连起来读）"。这使得谜题从4星难度降到了1星。

**建议**:

- 移除诗歌末尾的直接提示
- 让玩家通过群聊中的铺垫（"每行第三个字连起来读"）自行联想
- 如果担心玩家错过群聊线索，可以在提示系统中逐步引导

### 谜题5: 父亲文件夹第二层 (难度 2/5)

**线索链分析**:

```
铺垫: README.txt "密码是爸爸的灯熄灭的日子"
  |
推理: 阅读工作日志，找到关键日期
  |
解密: 20151123
  |
奖励: 《给夏灯.txt》完整信件
```

**问题**: 如前所述，README.txt 直接给出了密码 `20151123`，没有推理过程。

**额外问题**: "灯熄灭的日子"这个暗示指向什么？设计文档中是"签署第一份虚假证明的日子"（2015.11.23），但"灯熄灭"更容易让人联想到林晓宇坠亡日（2016.02.15）或夏父确诊癌症日。暗示的语义不够精确。

**建议**:

- README.txt 只保留暗示，不直接给出密码
- 将暗示改为更明确的表述，如"密码是我第一次背叛良心的日子"
- 在工作日志中强调 2015.11.23 的特殊性（如加粗、反复提及）

## 四、提示系统评估

### 当前实现

`PasswordDialog.jsx` 和 `QZone.jsx` 都实现了基于尝试次数的渐进式提示:

- 达到阈值后解锁提示按钮
- 辅助谜题10次后可跳过
- 显示"已尝试X次 / 再尝试Y次解锁提示"

**问题**:

1. `puzzle_hints.json` 的内容全部错误（密码、线索来源都与实际不符）
2. 提示阈值统一为 5/10/15/20，没有区分核心谜题和辅助谜题（设计文档中核心谜题是 5/8/12/15/20，辅助谜题是 3/5/10）
3. 没有"卡关超过10分钟触发陈默邮件"的机制
4. 没有"无操作5分钟弹出提示"的机制

### 与引导模式提议的关系

`提议:引导模式.md` 提出了认知节点系统和陈默邮件引导，`解密建议2.0.md` 提出了更详细的节点锁定和触发机制。这两个提议与当前提示系统可以互补:

- **渐进式密码提示**（当前已实现框架）: 适合密码输入场景
- **认知节点引导**（提议中）: 适合整体流程控制
- **陈默邮件提示**（提议中）: 适合卡关时的叙事性引导

**建议**: 三者结合使用，但优先修正 `puzzle_hints.json` 的内容。

## 六、优先修复建议

### P0: 必须立即修复

1. **重写 `puzzle_hints.json`**: 所有密码和线索内容与设计文档对齐
2. **修正 `photo_001_exif.json`**: 相机改为 Canon EOS 550D，参数改为 f/2.8, 1/125, ISO400
3. **修改 `README.txt`**: 移除第二层密码的直接展示，改为暗示性表述

### P1: 核心体验缺失

1. **实现隐写文字机制**: 在加密日志预览区添加白色隐藏文字，支持选中显示
2. **在 `encrypted_diary.json` 中添加 preview 字段**: 包含可见文字和隐藏文字
3. **在 `albums.json` 中添加相册描述**: 包含密码提示文字
4. **区分提示阈值**: 核心谜题（5/8/12/15/20）vs 辅助谜题（3/5/10）

### P2: 体验优化

1. **移除诗歌末尾的直接提示**: "（每行第三个字连起来读）"应该由玩家从群聊线索中自行发现
2. **实现卡关检测**: 无操作超时后触发提示
3. **添加相册封面图**: 当前 `coverImg` 为空，影响视觉体验

## 七、总结

解密设计在叙事层面达到了很高的水准——每个密码都有情感意义，每次解锁都推动剧情，难度曲线的设计意图也很合理。但当前实现与设计之间存在严重脱节:

1. `puzzle_hints.json` 是错误的旧版本，会误导玩家
2. 最精巧的隐写文字机制完全未实现，导致加密日志谜题无法正常游玩
3. README.txt 直接暴露了两个密码，消除了推理乐趣
4. EXIF数据与设计不一致，破坏了相册密码的线索链

修复这些问题后，解密体验将与叙事设计形成良好的协同，真正实现"解密过程就是理解角色的过程"这一设计理念。
